# 实验 6-7：Agent 任务的端到端成本分析

配套《深入理解 AI Agent》第 6 章「实验 6-7 ★：Agent 任务的端到端成本分析」。

对一个典型的多轮 Agent 任务（客服退款）做**全链路成本拆解**，用**自建的轻量 tracing / 可观测系统**记录每次 LLM 调用的输入/输出/缓存 token、时延与成本，按步骤聚合出「哪一步最贵」；再做 **A/B 对比**，量化 **KV-cache 友好设计 + 上下文压缩** 对 token 与成本的真实影响。

- 只依赖 **OpenAI**（`gpt-4o-mini`），通过 openai Python SDK 调用。
- KV-cache 的节省是**真实**的：利用 OpenAI 的自动 prompt caching（前缀 ≥ 1024 token 且命中近期相同前缀时，`usage.prompt_tokens_details.cached_tokens > 0`，这部分输入按缓存价 5 折计费）。

## 文件

| 文件 | 说明 |
|------|------|
| `config.py` | 模型与价格（gpt-4o-mini 单价），token→成本换算函数 |
| `tracer.py` | 自建轻量 tracing：包裹每次 LLM 调用，记录 token/缓存/时延/成本，按步骤聚合并打印拆解表 |
| `agent.py` | 多轮客服退款 Agent 任务；提供两种上下文构造策略（朴素 vs KV缓存+压缩） |
| `demo.py` | 入口：跑单次任务成本拆解 + A/B 对比表 |
| `requirements.txt` / `env.example` | 依赖与环境变量示例 |

## 运行

```bash
pip install -r requirements.txt
export OPENAI_API_KEY=sk-...        # 你的 OpenAI key（gpt-4o-mini）
python demo.py
```

`demo.py` 会真实调用 OpenAI，约几十次 chat completion，运行一两分钟。

## A/B 两种策略

同一个 8 轮客服退款任务（查订单 → 查物流 → 查退款政策 → 查知识库 → 风控 → 发起退款 → 通知 → 关单），两组做的是**同样的逻辑工作**，只在上下文构造上不同：

- **A 朴素（无缓存/无压缩）**：
  1. 前缀不稳定——每轮在 system 最前面塞一个随机 session 头，破坏前缀一致性 → prompt cache 完全命中不了（`cached_tokens=0`）。
  2. 不压缩——每轮都把全部啰嗦的历史工具返回原样带上，上下文线性膨胀。
- **B 优化（KV-cache 友好 + 上下文压缩）**：
  1. 稳定长前缀——system（一份 > 1024 token 的系统规范 + 工具手册）逐字节不变地放在最前面 → OpenAI 自动缓存，后续轮次 `cached_tokens` 显著 > 0，按缓存价计费。
  2. 上下文压缩——只保留最近 2 轮的完整工具返回，更早轮次替换成一句话摘要，控制上下文增长速率。

> 为聚焦「输入侧」两个杠杆，两组都用 `temperature=0` 且限制输出长度（`max_tokens=160`），让输出 token 成本对两组是近似相等的固定项，避免模型生成长度的随机波动干扰对比。
>
> 工具环境是「受控」的（工具返回内容预设，真实系统里来自订单/物流/知识库后端），但**每一次 LLM 调用、每一份 token 用量、每一分成本都是真实打到 OpenAI 得到的**，保证可复现。

可以看到：朴素组 A 的输入 token 随轮次从 810 一路涨到 3460（上下文累积效应，最后一步最贵）；优化组 B 的输入 token 被压缩策略压住（末轮 2324 而非 3460），且从第 3 轮起持续命中 1024 缓存 token。

### (b) A/B 对比

```
===== A/B 成本对比（同一个 8 轮客服退款任务）=====
方案                             总输入tok      缓存tok      缓存率    输出tok       总成本($)
-------------------------------------------------------------------------------
A 朴素(无缓存/无压缩)                   18357          0     0.0%      937     0.003316
B 优化(KV缓存+压缩)                   15936       6144    38.6%     1146     0.002617
-------------------------------------------------------------------------------
节省量化:
  总 token:   A=19294  →  B=17082   减少 2212 (11.5%)
  缓存 token: A=0  →  B=6144
  总成本:     A=$0.003316  →  B=$0.002617   降低 $0.000699 (21.1%)
  成本倍率:   A 是 B 的 1.27 倍
```

结论：
- **压缩杠杆**：B 的输入 token 比 A 少约 13%（15936 vs 18357），因为旧轮次被摘要替换、上下文不再线性膨胀。
- **KV-cache 杠杆**：B 命中 6144 个缓存 token（缓存率 38.6%，A 为 0），这部分按 5 折计费。
- 单看**输入侧成本**，B 相对 A 降低约 30%，与书中「KV Cache 可降低 30%-60% 输入 token 成本」的经验区间一致；叠加两个杠杆后**端到端总成本降低 21%**（A 是 B 的 1.27 倍）。

## 注意事项

- 具体数字每次运行会有小幅波动：OpenAI 的 prompt cache 是**尽力而为**的（按约 128 token 的块缓存、约 5–10 分钟过期、偶尔未命中），因此某些轮次可能出现 `cached_tokens=0`（如上面的 turn-2/turn-4）。`demo.py` 在正式计量前会先跑一次「预热」把稳定前缀写入缓存，让命中更稳定。
- 价格写在 `config.py`，默认是 gpt-4o-mini 的公开单价（输入 \$0.15 / 缓存 \$0.075 / 输出 \$0.60 每百万 token）。换模型请同步改价格。
- 只使用 OpenAI；不要接入 OpenRouter / Anthropic / DeepSeek / SiliconFlow。
- 缓存命中的前提是稳定前缀 ≥ 1024 token——本项目的系统提示（系统规范 + 工具手册）刻意写到这个量级，这也是真实工程里「把系统提示和工具定义放在最前面、保持稳定」的原因。
